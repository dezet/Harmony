# React + WebSockets Frontend Plan Set Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace Harmony's Phoenix LiveView UI with a React (TypeScript) SPA served same-origin by Phoenix, using Phoenix Channels for the real-time dashboard and a JSON API (with new project CRUD) for reads/writes.

**Architecture:** The work is split into four ordered phase plans. Each produces a working, tested increment with its own commits. The React app is built by Vite into `elixir/priv/static/app/` and served by Phoenix. During Phases 0–2 the SPA is mounted under `/app` while LiveView stays the default at `/`, so `main` stays shippable; Phase 3 cuts over to the SPA at `/` and removes LiveView. Real-time data flows from the existing `Presenter.state_payload/0` through a new `ObservabilityChannel` that pushes snapshots into the React Query cache.

**Tech Stack:** Backend — Elixir ~> 1.19, Phoenix 1.8, Bandit, Ecto/Postgres, Phoenix.Channel, ExUnit. Frontend — TypeScript, Vite, React Router v6, the official `phoenix` Channels client, React Query (TanStack Query v5), React Hook Form + Yup, shadcn/ui (Tailwind + Radix, default theme), Vitest + React Testing Library, Playwright (e2e).

## Conventions (codebase facts the engineer needs)

- **OTP app:** `:symphony_elixir`. Core modules: `SymphonyElixir`, `SymphonyElixirWeb`. The application starts in `elixir/lib/symphony_elixir.ex` (`SymphonyElixir.Application.start/2`).
- **The HTTP server is started by `SymphonyElixir.HttpServer`** (a supervised child), NOT by `mix phx.server`. It calls `Endpoint.start_link()` with `server: true` and a runtime port from `SymphonyElixir.Config.server_port()`. The base endpoint config in `elixir/config/config.exs` has `server: false` and `check_origin: false`. `Plug.Static` added to the endpoint module works regardless of this startup path.
- **There is no `config/dev.exs` or `config/runtime.exs`.** All endpoint config lives in `config/config.exs` and is augmented at runtime by `HttpServer`.
- **The dev server port** is whatever `SymphonyElixir.Config.server_port()` resolves to. Find it before configuring the Vite proxy: `cd elixir && mix run -e 'IO.puts(SymphonyElixir.Config.server_port())'`. The Vite proxy target uses that port.
- **All frontend code lives in `elixir/assets/`** (Vite root) and builds to `elixir/priv/static/app/`.
- **`mix build` is `escript.build`** (the `symphony` CLI), unrelated to assets. Asset builds use the new `mix assets.build` alias (added in Phase 0).
- **Module/topic facts:** PubSub server is `SymphonyElixir.PubSub`; the dashboard PubSub topic is `"observability:dashboard"`; the update message is the bare atom `:observability_updated` (`elixir/lib/symphony_elixir_web/observability_pubsub.ex`). The snapshot builder is `SymphonyElixirWeb.Presenter.state_payload/0`.

## Cross-Plan Rules

- **Single serialization path:** the `ObservabilityChannel` and `GET /api/v1/state` MUST return the identical `Presenter.state_payload/0`. Never build a second snapshot serializer. If a value needs JSON normalization, normalize it once in `Presenter`.
- **Keep `main` shippable:** Phases 0–2 must not change behavior at `/` (LiveView stays default). The SPA is reachable only under `/app` until the Phase 3 cutover.
- **Do not remove any LiveView module before Phase 3.** Phases 0–2 are additive on the backend.
- **shadcn/ui first:** lean on the default shadcn theme; never write a custom component before checking whether shadcn provides one. `assets/src/components/ui/*` are generated by the shadcn CLI and are not hand-edited.
- **No `fetch` in components:** all network access goes through `assets/src/lib/api.ts` and React Query hooks, or the Channel hook.
- **TDD + frequent commits:** every behavioral change starts with a failing test (ExUnit for Elixir, Vitest for TS). Scaffolding/config steps use explicit verification commands with expected output. Commit at the end of each task.

## Global Validation

Backend, run from `elixir/` after each backend-touching task:

```bash
cd elixir
mix format --check-formatted
mix test
```

Frontend, run from `elixir/assets/` after each frontend-touching task:

```bash
cd elixir/assets
npm run lint
npm run test -- --run
npm run build
```

Before calling the whole goal done (Phase 3 complete), from `elixir/`:

```bash
cd elixir
make all
```

Expected: every command exits 0. If local Postgres is unavailable, record the exact failure and run the targeted non-DB tests for the task instead.

## Manual / E2E Gates

- **End of Phase 0:** `mix assets.build` succeeds and `GET /app` returns the built `index.html` (verified by a controller test and a manual `curl`).
- **End of Phase 1:** with the server running, open `/app`, confirm the dashboard renders live data and that triggering `POST /api/v1/refresh` (or any orchestrator state change) pushes an update without a page reload.
- **End of Phase 2:** create and edit a project from `/app/projects/new` and `/app/projects/:id/edit`; confirm it persists (visible in `GET /api/v1/projects` and the projects list).
- **End of Phase 3:** the Playwright e2e harness (`make e2e`) drives the React UI at `/`; LiveView routes and the legacy asset routes are gone (404 / removed); `make all` exits 0.
